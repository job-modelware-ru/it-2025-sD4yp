# Описание архитектуры программной системы

## Проект: Система рекомендаций в игровой индустрии

---

## 1. Общее описание системы

Разрабатываемая система представляет собой онлайн-сервис для игровой индустрии, предназначенный для разработчиков игр и игроков.  
Разработчики могут публиковать информацию о своих играх, а игроки — искать игры, ставить оценки, оставлять комментарии и получать персонализированные рекомендации.

Основная цель архитектуры — обеспечить минимальный, но функционально завершённый набор возможностей, достаточный для демонстрации принципов проектирования информационных систем и последующей реализации в рамках учебного проекта.

---

## 2. Архитектурный стиль

Система построена в соответствии с **клиент–серверной архитектурой**.

- Клиентская часть реализована в виде веб-интерфейса, доступного через браузер.
- Серверная часть представляет собой централизованное backend-приложение.
- Все данные хранятся в одной реляционной базе данных.

Backend логически разделён на отдельные компоненты, каждый из которых отвечает за свою бизнес-логику. Такое разделение упрощает сопровождение системы и повышает её читаемость с архитектурной точки зрения.

---

## 3. Компоненты системы

### 3.1 Клиентская часть (Web UI)

Web UI является основной точкой взаимодействия пользователей с системой и предоставляет следующий функционал:

- регистрация и аутентификация пользователей;
- просмотр каталога игр;
- публикация информации об играх (для разработчиков);
- выставление оценок играм (от 1 до 5);
- добавление комментариев;
- настройка предпочтительных жанров;
- получение персонализированных рекомендаций.

Взаимодействие с серверной частью осуществляется по протоколу HTTP/HTTPS.

---

### 3.2 Серверная часть (Backend)

Серверная часть размещается на одном сервере приложений и включает следующие логические компоненты:

#### Auth Service  
Отвечает за аутентификацию и авторизацию пользователей, включая регистрацию и вход в систему.

#### User Service  
Управляет профилями пользователей и хранит их настройки, включая предпочтения по жанрам игр.

#### Game Service  
Обрабатывает данные об играх: описание, жанры, системные требования и другую метаинформацию.

#### Rating Service  
Отвечает за хранение и обработку пользовательских оценок и комментариев к играм.

#### Recommendation Service  
Формирует персонализированные рекомендации игр на основе:
- жанровых предпочтений пользователя;
- ранее выставленных оценок.

---

## 4. Хранение данных

Все данные системы хранятся в **реляционной базе данных**.  
В базе данных содержится следующая информация:

- учетные записи пользователей;
- профили пользователей и их предпочтения;
- информация об играх;
- оценки и комментарии пользователей.

Серверная часть системы имеет прямой доступ к базе данных и взаимодействует с ней через ORM или аналогичный механизм.

---

## 5. Размещение системы (Deployment)

Система развёртывается с использованием трёх основных узлов:

- **Устройство пользователя** — персональный компьютер с веб-браузером;
- **Сервер приложений** — содержит все backend-компоненты системы;
- **Сервер базы данных** — содержит реляционную базу данных.

Взаимодействие между узлами осуществляется следующим образом:
- клиент и сервер приложений обмениваются данными по HTTP/HTTPS;
- сервер приложений взаимодействует с сервером базы данных через протокол доступа к БД.

---

## 6. Основной сценарий взаимодействия

Типичным сценарием работы системы является получение рекомендаций игр:

1. Пользователь отправляет запрос на получение рекомендаций через Web UI.
2. Recommendation Service запрашивает предпочтения пользователя и его оценки.
3. На основе полученных данных выбираются подходящие игры.
4. Список рекомендованных игр возвращается в клиентское приложение и отображается пользователю.

---

## 7. Заключение

Предложенная архитектура является простой, понятной и достаточной для реализации основных функций системы рекомендаций в игровой индустрии.  
Она позволяет продемонстрировать принципы модульного проектирования, разделения ответственности и построения клиент–серверных приложений в рамках учебного проекта.
